# How the TeamMatch Recommender Works

TeamMatch recommends **which engineers to staff onto which projects**. It combines a
**hybrid recommender** (the per-person scoring model) with a **portfolio optimizer**
(the layer that turns scores into a feasible, organisation-wide staffing plan).

This document explains the data, the scoring model, the optimizer, the evaluation, and
the exact **triggers** (what computation fires on which event). The last section maps
everything onto the course units and is honest about what is and isn't a trained ML model.

research, not a recommender concept). The course's headline ML model — **matrix
factorization** — is the natural next addition (§8).

## 2. Data model

Three CSVs in `data/` (and mirrored in `frontend/public/data/`), generated by
`src/data_generation/generate_dataset.py` (seed = 42, fully reproducible).

**`employees.csv`** (800) — the "users": `employee_id, name, level (L3–L8), role_category,
years_experience, primary_domain, skills [{skill, proficiency 1–5}], tech_tags,
personality_* (Big Five OCEAN), education, previous_companies, past_projects_count,
avg_past_performance, primary_location, timezone_offset, can_work_across_timezones,
current_staffed, available_from, diversity_group`.

**`projects.csv`** (300) — the "items": `project_id, title, description, domain, priority,
required_team_size_{min,target,max}, required_roles [{role, min_level, count}],
required_skills [{skill, min_proficiency, weight}], tech_requirements, duration_weeks,
target_start_date, status (completed | active | pipeline), current_staffing_count,
current_staffed_ids`.

**`historical_assignments.csv`** (2,200) — the **rating matrix** (sparse: 2,200 of
800×300 ≈ 0.9% of cells): `employee_id, project_id, role_on_project, level_at_time,
pm_review, peer_collaboration_avg, delivery_score, was_successful, true_effectiveness`.
- The three review columns are **noisy observed ratings** — what the recommender trains on.
- `true_effectiveness` is the **clean ground truth** — stored *only* so we can evaluate
  honestly (it is never used as a model input).

> **Note on `current_staffed_ids`:** the generator currently writes `"[]"` for every
> project, even when `current_staffing_count > 0`. That's a known data gap — see §8; it
> blocks the "staff understaffed *active* projects" mode until populated.

```

        │ loadEmployees / loadProjects / loadHistorical (PapaParse)    │
        │ → evaluateModel()  → "Model validity r"                      │

        └──────────────────────────────────────────────────────────────┘
                                  │
            ┌──────── user clicks "Run Recommender" ────────┐
            ▼                                                │
   (1) SCORE every (available engineer × pipeline project)  │  triggers
       pair → computeMatchScore()                           │  re-run
            ▼                                                │
   (2) OPTIMIZE → optimizeAssignments()                      │
       greedy → repair → cohesion-aware local search         │
            ▼                                                │
   (3) VALIDATE constraints → validateAssignments()          │
            ▼                                                │
```

**Trigger reference**

| Event | What fires |
|---|---|
| Page load | Load 3 CSVs → `evaluateModel()` (ground-truth correlation) → seed pipeline |
| **Run Recommender** | Score all pairs → optimize → validate → render → `console.log("[TeamMatch] …")` diagnostics |

| Add / Remove project | Mutates pipeline, clears results (forces a fresh run) |
| Reset | Restores the original 6 pipeline projects |

## 4. The scoring model (the recommender)

For each `(employee, project)` pair, `computeMatchScore()` (in `frontend/src/lib/scorer.ts`)
produces five 0–1 signals and blends them with **segment-specific weights**.

**Signals**
1. **Skill match (content-based)** — for each required skill,
   `min(empProficiency / max(reqMinProficiency, 1), 1.3)`, weighted by the skill's weight,
   averaged, capped at 1.
2. **History (collaborative / track record)** — average of `(pm_review +
   peer_collaboration_avg + delivery_score)/3` over the employee's **same-domain** past
   assignments, normalised to 0–1; `0.5` if no relevant history.
3. **Personality fit** — `((O + C + E + A + (6 − N))/5 − 1)/5` (neuroticism inverted).
4. **Level / role fit** — `1.0` if the employee meets the level *and* role, `0.7` if level
   only, `0.3` if under-qualified.
5. **Novelty** — `1.0` if the employee's domain differs from the project (exploration),
   `0.4` if it matches.

**Segment routing (the "triggers" inside scoring)** — `getSegmentInfo()` computes a smooth
seniority signal `0.5·(levelIndex/5) + 0.5·min(pastProjects/8, 1)`:

| Seniority | Segment | Weights (skill / history / personality / level / novelty) |
|---|---|---|
| `< 0.33` | **exploration** (juniors) | 0.40 / 0.10 / 0.25 / 0.10 / **0.15** |
| `0.33–0.60` | **balanced** | 0.35 / 0.35 / 0.20 / 0.10 / 0 |
| `> 0.60` | **exploitation** (seniors) | 0.35 / **0.40** / 0.15 / 0.10 / 0 |

Final score = `clamp(weightedSum × 10, 1, 10)` (one decimal). Juniors are rewarded for
exploration/serendipity; seniors are rewarded for proven track record.

## 5. The optimizer (scores → a staffing plan)

`optimizeAssignments()` (in `frontend/src/lib/optimizer.ts`) turns the score matrix into a
portfolio-wide assignment. **This is the part that is *not* a classic recommender** — it's
constrained combinatorial optimization.

1. **Global greedy** — build one unified list of *every* `(project, employee)` pair, sort
   by score **descending**, and walk it once, assigning each person to the first project
   where they fill an open required role. Sorting globally (not per project) is what makes
   it a *portfolio* optimizer.
2. **Repair** — fill any still-open required role slots from the remaining pool.
3. **Cohesion-aware local search** (8 passes, top-50 shortlist per project) — repeatedly try
   to swap a member for an unassigned candidate who fills the **same role** and raises the
   team objective:
   `objective = Σ(individual scores) + 1.2 × cohesion × teamSize`, where **cohesion** is the
   average pairwise **cosine similarity** of team members' Big-Five vectors. Team chemistry
   is therefore genuinely optimised, not just displayed.

**Hard constraints enforced:** team size `[min, max]`, required roles + minimum levels,
availability (`available_from ≤ today`, not `current_staffed`), no double-staffing.
**Soft constraint:** a quality floor (score ≥ 4.0) once a team has its minimum bulk.

**The thermometer (achieved vs. ceiling).** `portfolioUpperBound()` computes the score each
project could reach if it got its *dream team* with the no-double-staffing rule relaxed
(the same star can be counted for several projects). No feasible plan can beat it, so
`efficiency = achieved / ceiling`. 100% means there was **no talent contention** (e.g. a
single project, or a pool large enough that projects never compete). The console logs
`top-hire contention: N/total projects` so you can see exactly why.

## 6. Evaluation

`evaluation.ts` re-scores every historical pair with the live model and compares to the
stored `true_effectiveness`:
- **Model validity `r`** — Pearson correlation between predicted score and ground truth.
- **Top-decile precision** — of the model's top-10% picks, how many were genuinely top-tier.
- **Coverage** — % of target slots filled, teams fully/partially/un-staffed, average cohesion.

This is the course's *"evaluation beyond accuracy"* idea: we measure not just rating error
but whether the *recommended* people would actually have performed.

---

## 7. Performance note (why it's so fast)

A run scores `projects × pool` pairs (e.g. 6 × 70 = 420), sorts them, and does a few
local-search passes — about **8–12 ms**. Fast is *expected* at this scale; open
DevTools → Console and the `[TeamMatch] …` line reports pairs scored, score spread, greedy
assignments, cohesion swaps, contention, achieved/ceiling and elapsed-ms on every run.

## 8. Honest positioning & roadmap

**Implemented (course units 3–5):** content-based filtering, a memory-based collaborative
signal, a hybrid blend, cosine-similarity cohesion, and proper evaluation.

**Not yet implemented (course unit 6 — the trained ML model): Matrix Factorization.**
Today the "collaborative" signal is a simple domain-scoped average, not a learned model.
To add the real thing, as taught (learn latent factors `P`, `Q` from the sparse rating
matrix by minimising squared error — `TruncatedSVD` / `NMF` / `ALS` in scikit-learn):
- Build a rating matrix from `historical_assignments` — recommended axis is
  **employee × domain** (or employee × skill) rather than employee × project, because new
  pipeline projects are *cold-start items* (MF can't score an item with no ratings — a
  limitation the deck calls out). Domain/skill factors generalise to brand-new projects.
- Train offline in Python, export predicted affinities, and feed them in as the
  collaborative component of the hybrid score.

**Two-mode recommender (proposed).** Run not only on **pipeline** projects but also on
**active** projects that are *understaffed* (`current_staffing_count < target`): staff only
the *gap*, treating already-assigned members (from a populated `current_staffed_ids`) as
fixed so cohesion is computed against the real existing team. This needs the
`current_staffed_ids` data gap from §2 fixed first.
